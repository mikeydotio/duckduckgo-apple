▶ create_release_branch start_new_release
Runnermacos-26-xlarge
Timeout10 minutes
Needscheck_automatic_freeze

Steps

  1. Checks out repo with full history (fetch-depth: 0) and all submodules
  2. Asserts that the workflow is running on the main branch
  3. Sets up Ruby and Bundler dependencies
  4. Runs start_new_release with platform:macos, optional version input, github_handle, and target_section_id

What start_new_release does

  1. Determines the new version: bumps the minor version number, or uses the provided version input
  2. Creates release/macos/<version> branch from main and pushes it
  3. Updates embedded files (e.g. privacy config) on the branch
  4. Updates Configuration/Version.xcconfig with the new version
  5. Calls find_release_dri to determine the Asana user to assign the task to
  6. Creates a new Asana release task from template (ID: 1206127427850447), assigned to the release DRI
  7. Updates the Asana release task with the commit log since the last public release (tasks included in this release)

Outputs

OutputDescription
release_branch_nameName of the newly created branch, e.g. release/macos/1.2.3
asana_task_idAsana task ID of the newly created release task
asana_task_urlFull URL of the Asana release task

⚠️ Branch Creation Failures

What can go wrong: The release branch already exists (from a previous failed run), or there is a version conflict.

Recovery: Check whether release/macos/<version> exists in GitHub. If it is stale, delete it. Then re-run the workflow.

▶ prepare_release
Callsmacos_release.yml — see Common Components ↗
Needscreate_release_branch, increment_build_number
Inputsasana-task-url (from create_release_branch), branch, commit-sha (from increment_build_number)

Purpose

Orchestrates the full build phase by calling the reusable macos_release.yml workflow, which fans out to two parallel builds:

  • DMG Build — calls macos_build_notarized.yml with release-type: release and create-dmg: true. Archives, notarizes, and uploads the DMG to S3.
  • App Store Build — calls macos_build_appstore.yml with destination: testflight. Builds and submits to TestFlight.

Asana Actions

asana_log_message — logs build status (debug symbols uploaded, DMG uploaded) to the Asana release task with template debug-symbols-uploaded and dmg-uploaded.

⚠️ Signing & Build Failures

What can go wrong: Code signing sync failure (the workflow has retry logic but can still fail), archive or notarization timeout.

Auto: asana_log_message logs build status to the Asana release task regardless of outcome.

Recovery: Check match repo SSH access. Verify that the MATCH_PASSWORD and SSH_PRIVATE_KEY_FASTLANE_MATCH secrets are current. Re-run the failed jobs in GitHub Actions.

⚠️ Tag & Merge Failures

What can go wrong: Merge conflicts between the release branch and main, or the tag already exists from a previous run.

Auto: asana_create_action_item with template internal-release-tag-failed — creates a subtask assigned to the release DRI with manual merge instructions. (Only fires on internal-release-bump context; not on code freeze.)

Recovery: Resolve conflicts locally on the release branch and push. Then manually trigger macos_tag_release.yml with the release branch name and the Asana task URL as inputs.

▶ publish_release
Callsmacos_publish_dmg_release.yml — see Common Components ↗
Needscreate_release_branch, tag_and_merge
Inputsasana-task-url, branch

Steps

  1. Downloads the tag artifact created by tag_and_merge
  2. Fetches and validates release notes from the Asana task
  3. Sets up Sparkle tools
  4. Fetches the DMG from S3
  5. Generates the Sparkle appcast entry using --release-to-internal-channel flag (internal channel only)
  6. Uploads updated appcast and DMG to S3 via upload_to_s3.sh
  7. Creates an Asana validation task via asana_create_action_item with template validate-check-for-updates-internal

Asana Actions

On success: creates a subtask with template validate-check-for-updates-internal (prompts DRI to verify the internal update channel). Uploads the appcast diff patch and old appcast XML as task attachments.

⚠️ Publish & Appcast Failures

What can go wrong: Appcast generation failure (malformed XML, missing DMG), S3 upload failure, or Asana update failure.

Auto: asana_create_action_item with template appcast-failed-internal — notifies the DRI that the appcast step failed and includes recovery instructions.

Recovery: Verify S3 credentials (AWS_ACCESS_KEY_ID_RELEASE_S3, AWS_SECRET_ACCESS_KEY_RELEASE_S3). Check appcast XML format. Manually trigger macos_publish_dmg_release.yml with inputs: asana-task-url, tag (from the tag_and_merge step), release-type: internal.

Internal Release Bump

Triggered automatically when code is pushed to a release branch. Validates conditions, runs tests, bumps build number, builds release, tags, and publishes to Sparkle.

File: macos_bump_internal_release.yml

Dispatch inputs: asana-task-url (optional string), base-branch (optional, default: main), skip-appstore (boolean, default: false)

Concurrency: macos-release (cancel-in-progress: false)

validate_input_conditions

prepare_release ↗
DMG Build
App Store Build (if skip-appstore != true)

▶ validate_input_conditions validate_internal_release_bump
Runnermacos-26
Timeout10 minutes

Steps

  1. Asserts the workflow is running on a release branch (or main)
  2. Finds or verifies the release task in Asana — searches the Apple Releases section (macOS) for a task matching the current marketing version; only considers tasks created in the last 5 days
  3. Checks the branch is NOT frozen — errors if a draft GitHub release named <version>+macos already exists (draft public release is the freeze indicator)
  4. Verifies code changes exist since the last tag — ignores changes under .github/, scripts/, fastlane/ directories; sets skip_release=true if no relevant changes are found
  5. Validates release notes are not empty and don't contain only placeholder text (<-- Add release notes here -->)
  6. Reports if an active hotfix task exists in the Apple Releases section — blocks normal bump and adds the release task assignee as a collaborator on the hotfix task

Outputs

OutputDescription
skip-releasetrue if no relevant code changes were found since the last tag; downstream jobs that depend on run_tests are skipped
asana-task-idAsana task GID of the current release task
asana-task-urlFull URL of the Asana release task
release-branchName of the release branch, e.g. release/macos/1.2.3
skip-appstoreForwarded from dispatch input or determined automatically

⚠️ Validation Failures

Frozen branch: validate_internal_release_bump errors if a draft GitHub release named <version>+macos already exists. This means a public release has been decided. No further internal bumps are allowed until the public release is tagged (which removes the draft and unfreezes the branch).

Empty release notes: The bump is blocked. Someone must fill in release notes in the Asana release task before the internal bump can proceed.

No code changes: skip-release is set to true and the workflow exits gracefully — this is NOT an error; it's expected when only non-code files changed (e.g. workflow configs, scripts, fastlane files).

Active hotfix: If an active hotfix task is found in the Asana Apple Releases section, the normal internal bump is blocked to prevent conflicts with the in-progress hotfix. The release task assignee is automatically added as a collaborator on the hotfix task.

Purpose

Runs unit tests and static analysis checks on the release branch. Skipped entirely when no relevant code changes have been detected (i.e. when skip-release is true).

Key difference from Code Freeze

This job passes internal-release-bump: true to macos_tag_release.yml. This flag enables automatic action item creation on tag or merge failure — an Asana subtask is created using the internal-release-tag-failed or merge-failed template to notify the release DRI with manual recovery instructions.

Auto: Because internal-release-bump: true is passed to tag_release, failures automatically trigger asana_create_action_item:

  • Merge to main failure → action item with template merge-failed

Phase 1 — Job Details

▶ create_release_branch start_new_release
Runnermacos-26-xlarge
Timeout10 minutes

Steps

  1. Checks out repo with full history (fetch-depth: 0) and all submodules
  2. Asserts the workflow is running on the main branch
  3. Sets up Ruby and Bundler dependencies (first time)
  4. Runs start_new_release with platform:macos, is_hotfix:true, and github_handle
  5. Sets up Ruby and Bundler dependencies again (after branch checkout, because the plugin version may differ on the hotfix branch)
  6. Adds an Asana comment using template hotfix-branch-ready to notify that the branch is ready for fixes

What start_new_release does (hotfix)

  1. Calls the GitHub API to find the latest public release tag (e.g. 1.105.0+macos)
  2. Bumps the patch version (e.g. 7.5.0 → 7.5.1)
  3. Creates a hotfix/macos/<version> branch from that release tag and pushes it
  4. Updates Configuration/Version.xcconfig with the bumped patch version
  5. Creates a new Asana hotfix task from template (ID: 1206724592377782), assigned to the release DRI

Note: Double Ruby Setup

Ruby dependencies are installed twice: before the fastlane run (to be able to invoke it) and again after it (because start_new_release checks out the new hotfix branch, which may have a different version of the plugin in its Gemfile.lock).

Callsmacos_tag_release.yml — see Common Components ↗

Key detail: merge target

The base-branch input controls where the hotfix is merged after tagging:

  • If current-internal-release-branch is provided, the hotfix merges there — so the ongoing internal release picks up the fix automatically.
  • If not provided, the hotfix merges to main.

The tag is created as a prerelease (prerelease: true). The actual public release is promoted during the manual Publish step.

⚠️ Merge Target Complexity

What to be aware of: If current-internal-release-branch is provided, the hotfix merges there — not main. This is intentional: it ensures the in-progress internal release incorporates the fix.

If merge fails: A merge-failed action item is created in Asana. Resolve the conflicts manually on the target branch, push, then manually trigger macos_tag_release.yml with the hotfix branch and Asana task URL as inputs.

Publishing the Hotfix (Manual Step)

After Phase 2 completes, the hotfix is tagged as a prerelease but NOT yet published to Sparkle. To publish:

  1. Go to GitHub Actions and manually dispatch macos_publish_dmg_release.yml
  2. Set asana-task-url to the hotfix task URL
  3. Set tag to the tag created by the tag_and_merge step (find it on the GitHub Releases page)
  4. Set release-type to hotfix

This triggers appcastManager.swift --release-hotfix-to-public-channel, publishes the DMG to the public Sparkle channel, and creates an Asana validation task. Variant creation runs automatically after publish.

⚠️ Publishing the Hotfix

The publish step is NOT automatic — you must manually trigger macos_publish_dmg_release.yml after Phase 2 completes.

Required inputs:

  • asana-task-url — the hotfix Asana task URL
  • tag — the tag created in tag_and_merge (get from the GitHub Releases page)
  • release-type — must be hotfix

After publish: variant creation runs automatically and an Asana validation task is created in the release task.

Release Branch Freeze Mechanism

The freeze mechanism prevents further internal releases once a version is ready for public release. Without it, any push to the release branch would trigger another internal bump, potentially overwriting the build that QA has already validated.

  1. Freezing. Triggered automatically by macos_promote_testflight.yml when freeze-release is true (the default). After promoting a TestFlight build to the App Store, the action freeze_release_branch runs and creates a draft GitHub release named <version>+<platform> (e.g., 7.5.0+macos). This draft release is never published — it exists purely as a machine-readable signal.
  2. Blocking. When validate_internal_release_bump runs on a push to the release branch, it calls GitHelper.assert_release_branch_is_not_frozen, which checks for the presence of that draft release. If found, it sets the skip_release GHA output to true and the workflow exits without building a new internal release.
  3. Unfreezing (automatic). When tag_release runs for a public release (is_prerelease: false), it calls GitHelper.unfreeze_release_branch before creating the real public tag. This deletes the draft release, unfreezing the branch as part of the normal public release flow.
  4. Unfreezing (manual). If you need to allow internal bumps again — for example, because the public release was delayed — go to GitHub → Releases and delete the draft release named <version>+<platform> manually.
Flow summary:
Promote to App Store → Draft release created (frozen) → Push to release branch skipped → Public release tags → draft deleted (unfrozen)

Sparkle / Appcast Pipeline

macOS uses the Sparkle framework for over-the-air updates. The pipeline generates and publishes an appcast XML file to S3 so that existing installs can discover new builds.

Sparkle Tools

The sign_update and generate_appcast command-line tools are downloaded at build time from the sparkle-project/Sparkle GitHub releases. They are used for signing DMG entries with the EdDSA private key and regenerating the appcast XML.

appcastManager.swift

A Swift script at macOS/scripts/appcast_manager/appcastManager.swift that manages appcast XML generation. It has three operating modes:

  • --release-to-internal-channel — used by the code freeze and internal bump flows. Adds the new DMG to the internal Sparkle channel (appcast URL for the internal config).
  • --release-to-public-channel — used for public releases. Promotes the already-published internal entry to the public appcast URL.
  • --release-hotfix-to-public-channel — used for hotfix releases. Publishes a new DMG entry directly to the public channel, bypassing the internal channel.

Publishing Flow

  1. Fetch the current appcast XML from S3.
  2. Back up the existing appcast (uploaded as a workflow artifact for rollback).
  3. Download the DMG from S3.
  4. Extract release notes from the Asana release task body using the fetch_release_notes helper. Notes must not be empty or placeholder — the publish step validates this before proceeding.
  5. Run appcastManager.swift with the appropriate mode to sign the DMG entry and generate the updated appcast XML.
  6. Upload the new appcast to S3, replacing the previous version.

Post-publish Validation

After a successful publish, an Asana action-item subtask is created using the validate-check-for-updates-internal or validate-check-for-updates-public template. The appcast diff and release notes are attached to the task for manual verification before the release is considered complete.

Appcast URLs

Appcast feed URLs are stored in macOS/Configuration/App/Sparkle.xcconfig under the pattern SPARKLE_URL_<CONFIG>. Release automation reads these values to determine the correct S3 target for each build configuration:

  • SPARKLE_URL_RELEASE — public appcast: staticcdn.duckduckgo.com/macos-desktop-browser/appcast2.xml
  • SPARKLE_URL_ALPHA — alpha appcast: staticcdn.duckduckgo.com/macos-desktop-browser/alpha/appcast-alpha.xml
